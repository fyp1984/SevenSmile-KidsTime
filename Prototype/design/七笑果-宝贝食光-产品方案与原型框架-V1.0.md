## 一、Point：结论先行

### 1.1 核心结论

- 七笑果宝贝食光（SevenSmile KidsTime）应定位为 `面向儿童家长的个性化营养饮食决策与执行助手`。
- 首个核心种子用户明确锁定为 `提出需求的奶爸本人`，并同步覆盖家庭主妇、祖辈照护者等家庭实际执行角色。
- 产品首期必须围绕 `四大营养基础模块 + 一大差异化互动增长引擎` 展开：`信息采集`、`方案生成`、`素材配套`、`动态跟踪`、`激励与亲子互动`。
- 微信小程序是首期最佳载体，适合承载轻表单、卡片式方案、家庭协同和低门槛传播。
- 七笑果的差异化，不是“再做一个育儿内容平台”，而是在权威营养主线之外，增加 `孩子愿意主动参与` 的奖励、游戏、许愿和亲子协作机制。
- 在动态跟踪模块中引入 `儿童电话运动手表运动数据接入`，有助于把“摄入建议”与“运动消耗”联动，提升方案个体化程度。
- 首期坚持 `免费获客 + 产品验证`，不引入复杂收费链路与重社群运营。

## 二、Reason：为什么这样判断

### 2.1 核心用户的痛点非常集中

首期目标用户不是没有爱心，而是缺少专业认知与持续执行能力，典型痛点包括：

- 不知道孩子当前年龄和体型对应的饮食重点是什么。
- 知道一些营养概念，但不会拆到早餐、午餐、晚餐。
- 工作忙，无法长期人工制定每周菜单。
- 面对挑食、过敏、口味偏好时，不知道如何替代。
- 拿到体检报告后，不知道饮食上该做什么调整。
- 希望家里不同照护者都能看懂并执行同一套方案。

### 2.2 产品价值必须同时覆盖“决策、执行、复盘”

如果只解决其中一个环节，产品很难持续使用：

- 只做内容，用户看完仍然不知道怎么买、怎么做。
- 只做菜谱，缺少个体差异与专业解释，信任感不够。
- 只做记录，用户要自己判断下一步怎么调整，负担仍在家长身上。

因此，七笑果首期必须形成闭环：

`孩子情况采集 -> 个性化生成 -> 每日执行素材 -> 数据回流迭代`

### 2.3 微信小程序比独立 App 更适合当前阶段

- 首次使用成本低，适合免费获客。
- 家长对微信生态的接受度高，分享孩子档案和菜单更自然。
- 适合高频查看今日餐单和随手录入数据。
- 小程序更利于先验证产品价值，再决定是否扩展为独立 App。

## 三、Evidence：产品定义与设计依据

### 3.1 用户角色设计

| 角色 | 描述 | 核心目标 | 典型使用场景 |
| --- | --- | --- | --- |
| 核心奶爸用户 | 首个种子用户，工作繁忙但主动照顾孩子饮食 | 快速拿到可靠、能执行的日常餐单 | 通勤途中查看、晚间采购、周末统一规划 |
| 家庭主妇 | 主要餐食执行者 | 获得清晰食材与做法安排 | 每日做饭前查看今日计划 |
| 祖辈照护者 | 经验丰富但营养观念可能不一致 | 看懂标准化建议并减少家庭争议 | 白天照护时按清单执行 |
| 家庭协作成员 | 另一位家长或辅助照护者 | 同步信息并分担采购/准备 | 家庭成员共享采购与提醒 |
| 6-12 岁儿童 | 能独立完成基础交互的孩子 | 通过任务、游戏、奖励获得参与感 | 领取任务、填写偏好、抽奖、许愿 |

### 3.2 用户痛点拆解

| 痛点 | 表现 | 现有替代方案缺陷 | 七笑果解决方式 |
| --- | --- | --- | --- |
| 认知不足 | 不知道该给孩子吃什么比例最合适 | 搜索碎片化内容，结论冲突 | 用权威规则自动生成适龄方案 |
| 执行困难 | 每周想菜单耗时且重复 | 临时决定、营养波动大 | 提供一周/月度方案和采购清单 |
| 个性差异复杂 | 挑食、过敏、口味偏好难兼顾 | 通用菜谱无法直接用 | 方案支持替代选项与禁忌过滤 |
| 缺少反馈闭环 | 体检数据和日常饮食没有连接 | 只能靠经验调整 | 用生长与体检数据驱动方案迭代 |
| 家庭协同低 | 家庭成员理解不一致 | 口头传达容易走样 | 小程序卡片化方案统一执行口径 |
| 孩子主动性弱 | 孩子把饮食安排看成“被要求做” | 同类产品大多只服务家长，不服务孩子 | 用盲盒、任务、许愿和亲子协作把孩子拉进闭环 |
| 摄入与消耗脱节 | 只知道吃了什么，不知道当天动了多少 | 碳水、蛋白和加餐安排容易失衡 | 接入儿童电话手表运动数据，形成摄入-消耗联动判断 |

### 3.3 核心价值主张

- `给家长一套今天就能执行的饮食方案`
- `每一份推荐都能说清依据和替代方案`
- `孩子数据有变化，方案就能自动迭代`
- `让孩子愿意主动参与健康饮食，而不是被动配合`
- `让家长同时看懂孩子每天吃了多少、动了多少、还应如何调整`

### 3.4 产品边界

首期明确不做以下能力：

- 在线诊疗或疾病诊断
- 替代医生或营养门诊判断
- 深度社群互动与内容社区
- 自营商城和复杂供应链系统
- 重视频课程体系

### 3.5 规则引擎设计原则

| 原则 | 说明 |
| --- | --- |
| 权威优先 | 规则基于膳食指南、DRIs、生长标准和专项食养指南 |
| 人群分层 | 按年龄段、体型状态、饮食禁忌、口味偏好生成方案 |
| 先规则后 AI | 先用确定性规则做推荐，再用 AI 生成解释与替代项 |
| 建议可解释 | 用户可查看“为什么这样安排” |
| 动态可调整 | 允许按执行情况、季节、体检结果更新 |

## 四、Proposal：完整产品方案

### 4.1 产品总体架构

```text
儿童档案采集
  -> 营养画像建模
  -> 周/月度饮食方案生成
  -> 每日采购清单与菜谱素材
  -> 执行反馈与体检数据回流
  -> 互动激励与亲子协作引擎
  -> 方案重新计算与风险提示
```

### 4.2 四大基础模块 + 差异化互动增长引擎

#### 4.2.1 模块一：用户信息采集

目标是把家长“模糊描述的孩子情况”转成可计算的营养画像。采集维度包括：

- 基础信息：年龄、性别、身高、体重、活动水平、就学情况
- 健康信息：过敏、乳糖不耐、便秘倾向、缺铁史、维生素 D 补充情况
- 饮食偏好：口味、可接受食材、讨厌食材、家庭饮食习惯
- 执行约束：预算、做饭时间、是否可午餐自带、是否可使用外卖/半成品

#### 4.2.2 模块二：个性化方案生成

系统根据儿童画像输出：

- `一周方案`：适合高频执行和快速开始
- `一月方案`：适合家庭做长期节奏规划
- `每日结构`：早餐、午餐、晚餐、加餐
- `替代版本`：普通日、上学日、周末日、食欲差日、户外活动日

生成逻辑遵循以下原则：

- 主食不单一，兼顾谷薯杂豆轮换
- 蛋白质来源多样化，禽畜鱼蛋奶豆类合理轮换
- 蔬果按颜色和种类做多样性覆盖
- 对过敏/禁忌食材自动过滤
- 对挑食儿童提供低阻力替代项

#### 4.2.3 模块三：素材配套服务

系统不只给方案，还要把“去做”这一步降到最低：

- 每日食材采购清单
- 每周采购汇总清单
- 食材储存建议
- 菜谱步骤链接
- 适合家长通勤或做饭时收听的饮食播客合集
- 渠道建议与采购提醒

#### 4.2.4 模块四：数据动态跟踪

系统通过持续回收数据完成“个体化”：

- 定期录入身高、体重、腰围、睡眠、运动量
- 脱敏上传体检报告核心指标
- 接入儿童电话运动手表的步数、运动时长、活动能量消耗、佩戴时长等数据
- 记录执行反馈，如吃完程度、孩子接受度、过敏反应
- 生成趋势图和风险提示
- 自动重算下周或下月方案

#### 4.2.5 模块五：激励与亲子互动引擎

这一模块是产品区别于同类产品的核心差异化能力，目标不是单纯“游戏化”，而是让孩子在饮食改善过程中产生期待感、成就感和亲子连接。

核心子模块包括：

- `盲盒奖励`：支持按日、周、月配置自动发放，奖励内容以“大餐券”“家庭共享餐”“心愿加权券”等为主。
- `在线抽奖`：采用可配置概率池，满足任务或积分消耗条件后参与，奖品必须来自已审核奖励池。
- `积分兑换`：孩子通过完成饮食任务、签到、连续打卡、亲子协作任务累计积分，用于兑换专属大餐奖励。
- `偏好采集`：支持孩子自主录入最喜欢的 3 类食物和最讨厌的 3 类食物，参与奖励推荐计算。
- `饮食小游戏`：通过食物分类、营养搭配、闯关答题等轻互动游戏，解锁奖励资格。
- `许愿池`：孩子可以提交“大餐愿望”，家长审核后将其纳入奖励池。
- `亲子协作`：家长查看孩子任务进度、手动追加奖励，并与孩子共同完成共享大餐任务。
- `家长配置中心`：奖励边界、许愿池审核开关、祖辈可否参与手动发奖均由家长自主配置。

#### 4.2.6 模块六：运动消耗数据接入

这一模块是动态营养推荐的重要增强能力，目标是把“当天活动量”纳入餐单微调，而不是只依据静态年龄和体征。

核心能力包括：

- `设备绑定`：支持绑定儿童电话运动手表账号或设备授权关系。
- `运动数据采集`：采集步数、活跃分钟数、活动时段、估算能量消耗。
- `消耗评估`：将运动数据转化为可用于饮食规划的日级活动系数。
- `营养联动`：对早餐、午餐、晚餐和加餐中的碳水、蛋白、荤素比例做轻量微调。
- `家长解释`：以易懂语言展示“今天活动量更高，因此午后加餐增加一份乳制品/主食替代项”。

设计原则如下：

- 首期优先做 `单品牌或单渠道灰度接入 + 手动补录兜底`，避免被设备开放能力卡住。
- 运动数据只用于营养建议，不输出医疗或运动能力诊断。
- 当设备数据缺失、异常或长时间未同步时，系统自动回退到基础营养模板。

模块设计原则如下：

- 奖励必须服务健康饮食目标，不能演变成高糖高油刺激。
- 所有奖励逻辑都要可解释、可配置、可审计。
- 所有儿童交互采用短路径、强反馈、低文本密度设计。
- 所有需要消费型决策的动作必须经家长确认或复核。
- 所有家庭权限边界优先遵循家长配置，而不是平台强行固定。

### 4.3 关键业务流程

#### 4.3.1 首次使用流程

`进入小程序 -> 品牌欢迎页 -> 创建孩子档案 -> 录入饮食画像 -> 选择计划周期 -> 生成首份方案 -> 查看今日餐单`

#### 4.3.2 日常执行流程

`首页查看今日餐单 -> 点开单餐详情 -> 查看食材与做法 -> 勾选已采购/已完成 -> 记录孩子接受度 -> 系统调整后续方案`

#### 4.3.3 周期复盘流程

`进入成长跟踪页 -> 上传身高体重/体检摘要 -> 查看趋势变化 -> 接收新一轮调整建议`

#### 4.3.3A 运动数据同步流程

`进入跟踪页 -> 绑定儿童电话手表 -> 同步近 7 日步数/运动时长/消耗 -> 系统计算活动等级 -> 回写到当日与次日饮食建议`

#### 4.3.4 互动激励流程

`孩子完成饮食任务或小游戏 -> 获得积分/盲盒/抽奖资格 -> 查看奖励中心 -> 提交许愿或兑换奖励 -> 家长审核确认 -> 奖励进入可兑现状态`

#### 4.3.5 亲子协作流程

`系统发布亲子协作任务 -> 家长与孩子共同完成 -> 双方确认打卡 -> 解锁共享大餐或家庭奖励 -> 进入亲子回顾`

#### 4.3.6 家长配置流程

`进入我的 -> 家长配置中心 -> 设置奖励边界 -> 设置许愿池是否默认审核 -> 设置祖辈是否可参与手动发奖 -> 全局生效并写入权限日志`

### 4.4 页面与导航架构

首期建议采用 `5 个一级导航`，其中把原先偏功能性的 `采购` 一级入口收敛为内容页入口，新增更具留存价值的 `乐园` 作为一级导航：

- `首页`
- `方案`
- `乐园`
- `跟踪`
- `我的`

| 页面 | 主要功能 | 入口 | 出口 |

| 方案页 | 一周/月度切换、每日卡片、替代方案 | 首页、底部导航 | 单日详情 |
| 单日详情页 | 早餐午餐晚餐加餐、营养说明、替换项 | 方案页 | 采购清单、反馈记录 |
| 采购页 | 日清单、周清单、渠道推荐 | 首页、单日详情、底部导航 | 外部链接、收藏 |

| 家长配置页 | 奖励边界、审核开关、祖辈权限 | 我的、亲子协作页 | 权限生效结果 |
| 跟踪页 | 数据录入、趋势图、风险提示 | 首页、底部导航 | 调整方案 |
| 设备连接页 | 绑定儿童电话手表、查看同步状态 | 跟踪页、我的 | 数据同步结果 |
| 体检上传页 | 脱敏上传与指标提取 | 跟踪页 | 跟踪结果 |
| 我的页 | 家庭成员、规则来源、隐私授权、反馈 | 底部导航 | 档案编辑、设置 |

### 4.5 微信小程序设计规范

#### 4.5.1 视觉基线

严格采用七笑果品牌规范，但对儿童营养场景做更温暖的表达：

- 主色渐变：`#FF9800 -> #E63900`
- 背景：`#FFF7ED -> #FFFFFF`
- 重点提示色：`#FFC107`
- 成功状态：`#52C41A`
- 风险提示：`#FA8C16`
- 主标题：`#4A2311`
- 正文：`#5F534D`
- 分割线：`#F3D9C6`
- 儿童互动强调色：`#7ED957`、`#6EC5FF`，用于游戏成功反馈与奖励状态

#### 4.5.2 组件规范

| 组件 | 规范 |
| --- | --- |
| 主按钮 | 大圆角、暖橙渐变、高对比白字 |
| 卡片 | `rounded-24px`，轻阴影，白底半透明 |
| 标签 | 胶囊样式，支持年龄段、过敏、口味筛选 |
| 图表 | 折线和环图优先，避免复杂大屏图形 |
| 表单 | 分步录入，每步不超过 6 个字段 |
| 底部导航 | 5 项以内，使用短文案和简洁图标 |
| 奖励组件 | 盲盒、奖券、积分币、任务徽章都采用大图标与卡片式表达 |
| 儿童按钮 | 文案不超过 6 字，优先动词，如“打开盲盒”“去闯关”“去许愿” |

#### 4.5.3 交互规则

- 单个可点击区域不小于 `44px`
- 关键信息一屏内可读，不堆长文案
- 每个页面只保留一个主任务
- 所有结论前置，解释折叠展示
- 默认展示“今日该做什么”，而不是系统背景
- 上传体检数据时必须先做脱敏引导
- 6-12 岁儿童页面优先使用图标、颜色和进度反馈，减少长文本阅读负担
- 奖励相关流程必须区分 `孩子可操作` 与 `家长确认` 两层权限

### 4.6 动态调整机制

#### 4.6.1 输入信号

- 新增身高体重
- 新增体检数据
- 最近 7 日平均步数
- 当日活跃分钟数
- 当日估算活动能量消耗
- 设备同步成功率与佩戴有效时长
- 最近一周执行完成率
- 最近一周孩子接受度
- 最近一周任务完成率
- 连续签到天数
- 积分累计与消耗情况
- 奖励领取与兑现状态
- 新增过敏或禁忌信息
- 季节切换与上学/放假场景变化

#### 4.6.2 调整策略

- 营养结构不变时，优先调整食材组合和烹饪形式
- 接受度下降时，优先推送等营养替代项
- 体重增长偏快或偏慢时，调整能量与加餐策略
- 缺铁、便秘、超重等风险提示时，切换到专项食养模板
- 当日活动消耗偏高时，优先增加优质碳水和蛋白补给建议
- 当日活动消耗显著偏低时，优先降低额外加餐和高能量替代项
- 奖励兑现率下降时，调整奖励池内容与触发门槛
- 孩子长期不参与时，优先推送低门槛小游戏和短周期任务

### 4.7 数据与技术落地建议

#### 4.7.1 推荐系统结构

- `规则层`：年龄段、体征、禁忌、口味、执行约束
- `知识层`：权威指南映射、食材库、营养标签库
- `计划层`：周/月度餐单、替代项、采购清单
- `反馈层`：执行数据、体检数据、趋势评估
- `设备层`：儿童电话手表绑定信息、同步记录、运动消耗摘要
- `激励层`：任务中心、签到、盲盒、抽奖、积分与奖励池
- `协作层`：家长审核、手动发奖、亲子共享任务
- `配置层`：奖励边界、许愿池审核策略、祖辈发奖权限、默认家庭权限模板
- `日志层`：用户操作日志、奖励发放日志、概率池命中日志、审核日志
- `解释层`：展示推荐原因、引用规则和注意事项

#### 4.7.2 推荐技术原则

- 首期优先采用 `规则引擎 + 配方模板 + 食材替代库`
- AI 用于：
  - 文案解释
  - 菜谱摘要整理
  - 播客与知识素材聚合
  - 用户问题的自然语言引导
- 运动联动首期采用 `设备数据摘要 + 活动等级规则映射 + 餐次微调模板`，避免直接依赖高复杂度实时算法
- 激励引擎采用 `规则配置 + 概率池配置 + 审核流` 的确定性架构，避免黑盒感
- 家庭权限采用 `家长配置优先` 原则，平台只提供默认建议模板

### 4.8 阶段规划

| 阶段 | 周期 | 目标 |
| --- | --- | --- |
| 阶段 0 | 1 周 | 规则口径冻结、原型确认、食材库和奖励池模型建模 |
| 阶段 1 | 2 周 | 完成建档、方案、采购、跟踪基础链路 |
| 阶段 2 | 2 周 | 完成盲盒、抽奖、积分、许愿池、亲子任务等互动模块 |

| 阶段 4 | 1-2 周 | 100 组家庭灰度试用与交互优化 |

### 4.9 成功指标

- 首次建档完成率 >= 70%
- 首次方案生成成功率 >= 95%

## 六、风险与控制

### 6.1 主要风险

- 规则过于复杂，导致首版交互负担过重
- 家长输入字段过多，首登流失
- 推荐结果过于学术化，不利于执行
- 体检数据上传流程过重，导致回填率低
- 儿童电话手表品牌分散，开放接口和数据质量差异较大
- 奖励机制设计不合理，可能导致“只为奖励而完成任务”
- 概率池和积分兑换缺乏透明度，易引发公平性质疑

### 6.2 控制策略

- 采用分步录入与默认值策略
- 首期只保留对推荐结果影响最大的字段
- 所有输出先给结论，再给专业说明
- 跟踪数据支持“手动填关键指标”和“脱敏上传”两条路径
- 运动数据采用“设备灰度接入 + 手动补录 + 异常回退模板”三层兜底
- 奖励规则在家长端显式展示触发条件、概率区间和审核状态
- 所有发奖、抽奖、兑换、审核动作保留完整日志用于复盘和申诉
- 奖励边界、默认审核开关、祖辈发奖权限均提供家长可配置项，并保留配置变更日志
