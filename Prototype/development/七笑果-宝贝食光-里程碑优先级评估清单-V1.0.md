## 1. Point：结论先行

### 1.1 核心结论

- 宝贝食光的产品内核必须收敛到一条最短闭环：`建档 -> 个性化方案 -> 每日执行 -> 数据回填 -> 动态调整`。
- 首期 `MVP` 不应追求“大而全”，重点是验证家长是否愿意持续使用这条主闭环，以及方案是否真的可执行。
- 中期应补齐“提高留存和差异化”的能力，包括互动增长、运动手表数据接入、家庭协作和更完整的动态规则。
- 后期再推进推广、商业化和生态扩展，如采购转化、会员服务、品牌合作和渠道放大。
- 若首期闭环验证失败，后续再强的互动和商业化设计都无法成立，因此资源应优先押注内核能力。

### 1.2 推荐决策

- 采用 `里程碑推进` 而不是 `功能堆叠` 的规划方式。
- 每个阶段只做与该阶段目标直接相关的能力，避免首期被非关键能力拖慢。
- 所有能力分为三类：
  - `内核能力`
  - `增长增强能力`
  - `商业化与生态能力`

### 1.3 主方案概览

| 阶段 | 目标 | 核心判断 |
| --- | --- | --- |
| M1：MVP 验证期 | 验证需求是否成立 | 优先证明家长愿意用、方案能执行、数据能回填 |
| M2：产品完善期 | 提升留存和差异化 | 补齐互动、设备接入、家庭协作与动态规则 |
| M3：推广与变现期 | 放大规模和收入 | 在已有留存基础上做渠道扩张、商业化闭环 |

## 2. Reason：为什么这样判断

### 2.1 当前最大风险不是功能不够多，而是主价值未被验证

对于宝贝食光来说，最关键的问题不是“能不能加更多模块”，而是：

- 家长会不会真的录入完整信息
- 生成的方案能不能真正落到餐桌
- 用户会不会在一周后继续回来
- 数据回填后，用户能不能感知到调整价值

如果这些核心问题没有被验证，后续的互动玩法、手表接入、商业化入口都只是附着在不稳定底座上的“锦上添花”。

### 2.2 互动与设备接入有价值，但不应抢占 MVP 的资源中心

互动增长和运动手表接入都很有差异化价值，但它们属于：

- 帮助提高留存和感知差异
- 帮助扩大个性化能力
- 帮助形成更强长期壁垒

它们并不是首个“用户愿不愿意用”的唯一前提，因此应放在主闭环验证后稳步上线。

### 2.3 商业化必须建立在可复用留存模型之上

后期如果要做：

- 采购转化
- 会员订阅
- 专家增值服务
- 品牌合作

前提一定是先证明产品的自然留存和长期价值，否则商业化会直接破坏用户信任。

## 3. Evidence：功能优先级拆解

### 3.1 功能分类

| 功能 | 分类 | 说明 |
| --- | --- | --- |
| 儿童档案建立 | 内核能力 | 所有推荐的前提 |
| 饮食画像采集 | 内核能力 | 决定方案个性化质量 |
| 一周/月度方案生成 | 内核能力 | 产品核心交付物 |
| 每日采购清单 | 内核能力 | 决定执行效率 |
| 菜谱/播客素材 | 内核能力 | 降低执行门槛 |
| 身高体重/体检跟踪 | 内核能力 | 决定动态调整价值 |
| 动态方案调整 | 内核能力 | 体现持续使用理由 |
| 家长查看解释依据 | 内核能力 | 建立专业信任 |
| 积分、盲盒、抽奖 | 增长增强能力 | 提高孩子参与和家庭留存 |
| 偏好采集、许愿池 | 增长增强能力 | 提升孩子接受度与参与感 |
| 亲子协作、手动发奖 | 增长增强能力 | 强化情感连接和家庭协同 |
| 儿童电话手表接入 | 增长增强能力 | 增强摄入-消耗联动能力 |
| 在线采购入口 | 商业化与生态能力 | 可转化但非 MVP 必需 |
| 会员订阅 | 商业化与生态能力 | 建立收入模型 |
| 品牌合作/营养商品推荐 | 商业化与生态能力 | 后期可扩展变现 |

### 3.2 产品内核定义

本产品的内核能力只有 5 项：

- `孩子信息建档`
- `个性化饮食方案生成`
- `采购与执行素材支撑`
- `跟踪与回填`
- `动态调整`

只有这 5 项被验证后，产品才算真正成立。

### 3.3 非首期不应重投入的能力

以下能力虽然有想象空间，但不应在 MVP 阶段重投入：

- 多品牌儿童电话手表全面接入
- 重互动游戏体系
- 复杂家庭权限模型
- 真实电商交易闭环
- 会员权益体系
- 私域社群运营工具

## 4. Proposal：里程碑优先级评估清单

### 4.1 M1：MVP 验证期

#### 4.1.1 阶段目标

验证用户最核心的需求是否成立：

- 家长愿不愿意完成建档
- 家长能否理解并执行今日/本周餐单
- 用户是否愿意回填身高体重、反馈和体检摘要
- 用户是否能感知“调整后的方案更有价值”

#### 4.1.2 必须做

| 功能 | 优先级 | 原因 |
| --- | --- | --- |
| 建档与饮食画像采集 | P0 | 没有画像就没有推荐基础 |
| 一周方案生成 | P0 | 最短路径验证核心价值 |
| 单日三餐详情 | P0 | 帮用户真正执行 |
| 采购清单 | P0 | 降低从“知道”到“去做”的阻力 |
| 菜谱/播客素材 | P1 | 增强执行便利性 |
| 身高体重与执行反馈回填 | P0 | 验证动态闭环 |
| 体检摘要上传 | P1 | 强化专业性，但可轻量先做 |
| 方案动态调整 | P0 | 构成持续使用理由 |
| 权威依据展示 | P0 | 建立家长信任 |

#### 4.1.3 建议暂缓

| 功能 | 原因 |
| --- | --- |
| 盲盒、抽奖、积分 | 先不作为首个价值验证的关键路径 |
| 许愿池、亲子协作 | 增强留存但非首要验证点 |
| 儿童电话手表接入 | 先在设计中预留，避免首期受设备生态拖慢 |
| 在线采购真实打通 | 先用清单与推荐链接替代 |
| 会员和付费体系 | 首期不应引入变现摩擦 |

#### 4.1.4 MVP 成功标准

- 建档完成率达到既定目标
- 首次方案生成成功率稳定
- 用户一周内有真实执行和反馈回填
- 用户能感知“方案有在变得更贴合”
- 至少形成一批愿意自传播的种子家庭

### 4.2 M2：产品完善期

#### 4.2.1 阶段目标

在主闭环成立后，提高留存、差异化和家庭协同能力。

#### 4.2.2 重点建设

| 功能 | 优先级 | 价值 |
| --- | --- | --- |
| 积分体系 | P0 | 建立可持续参与反馈 |
| 盲盒/抽奖/兑换 | P1 | 提高孩子参与和趣味性 |
| 偏好采集 | P0 | 提升奖励推荐命中率 |
| 许愿池 | P1 | 增强孩子期待感 |
| 亲子协作任务 | P1 | 增强情感连接与家庭参与 |
| 家长配置中心 | P0 | 控制奖励边界与审核权限 |
| 儿童电话手表单渠道接入 | P1 | 增强摄入-消耗联动能力 |
| 设备连接与活动等级规则 | P1 | 形成数据增强型差异化 |
| 更丰富的动态调整模板 | P0 | 提升个性化深度 |

#### 4.2.3 阶段重点不是“更多功能”，而是“更高复访”

这一阶段的核心问题是：

- 孩子是否更愿意参与
- 家长是否更愿意长期坚持
- 家庭是否更容易形成协同
- 产品是否形成明显区别于通用食谱工具的优势

#### 4.2.4 中期成功标准

- 孩子主动参与度显著提升
- 家长周留存率达到目标
- 乐园入口、任务、许愿池等互动功能具备稳定使用率
- 设备接入用户中，活动消耗联动建议有正向反馈

### 4.3 M3：推广与商业化期

#### 4.3.1 阶段目标

在留存模型成立后，开始做规模放大和收入模型验证。

#### 4.3.2 可进入的方向

| 方向 | 优先级 | 说明 |
| --- | --- | --- |
| 在线采购转化 | P1 | 从采购清单延伸到购买入口 |
| 会员订阅 | P1 | 提供高级模板、深度跟踪、专项计划 |
| 专家服务 | P2 | 如营养咨询、专题评估服务 |
| 品牌合作 | P2 | 与健康食品、餐具、儿童运动设备合作 |
| 家庭成长报告 | P1 | 可作为高级权益或会员功能 |
| 渠道推广与联名活动 | P1 | 放大获客与品牌影响力 |

#### 4.3.3 后期推进原则

- 商业化不得破坏专业信任
- 采购转化不得替代客观建议
- 会员权益要增强服务，而不是锁死基础能力
- 推广动作要建立在真实口碑和复购行为之上

### 4.4 不同阶段的资源投入建议

| 阶段 | 资源重点 |
| --- | --- |
| MVP | 产品定义、规则引擎、核心页面、基础数据回填 |
| 中期 | 增长玩法、设备接入、协作机制、数据分析 |
| 后期 | 商业化设计、渠道合作、品牌传播、增长投放 |

### 4.5 一页式优先级总表

| 能力 | M1 MVP | M2 完善 | M3 变现 |
| --- | --- | --- | --- |
| 建档与画像 | 必做 | 持续优化 | 保持稳定 |
| 方案生成 | 必做 | 持续优化 | 保持稳定 |
| 采购清单与素材 | 必做 | 持续优化 | 可接转化 |
| 跟踪与动态调整 | 必做 | 加强深度 | 形成高阶报告 |
| 激励互动 | 暂缓 | 重点建设 | 持续迭代 |
| 家长配置中心 | 暂缓或轻量 | 重点建设 | 保持稳定 |
| 儿童电话手表接入 | 预留 | 单渠道灰度 | 多品牌扩展 |
| 在线采购 | 预留 | 小范围验证 | 重点建设 |
| 会员订阅 | 不做 | 预研 | 正式验证 |
| 专家服务 | 不做 | 预研 | 选择性上线 |

### 4.6 风险控制与评审口径

- 任何新增功能都应回答一个问题：是否直接帮助当前阶段目标成立。
- 若不能直接服务阶段目标，应延期到下一里程碑。
- 每个阶段结束前必须做一次 `功能减法评审`，避免范围膨胀。
- 首期所有讨论都应回到一个判断：`这项能力是否直接帮助验证用户需求`。

## 5. 附录

### 5.1 推荐使用方式

- 产品评审会：用本文档判断功能是否进入当前版本
- 研发排期会：用本文档做 P0/P1/P2 切分
- 商业化规划会：用本文档区分“当前验证”和“后期变现”

### 5.2 当前建议结论

- 现在最该做的是 `M1 MVP 验证`
- 现在最该守住的是 `产品内核闭环`
- 现在最不该做的是 `过早商业化和过度功能扩张`
