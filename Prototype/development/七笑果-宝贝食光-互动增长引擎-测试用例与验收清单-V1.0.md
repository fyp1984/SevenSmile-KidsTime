## 1. Point：结论先行

### 1.1 核心结论

- 本模块测试重点不在单页面展示，而在 `奖励公平性`、`权限准确性`、`事务一致性` 和 `日志完整性`。
- 验收必须覆盖四层：`单元测试`、`集成测试`、`安全测试`、`100 组家庭灰度验证`。
- 上线前强制门槛建议采用：`0 阻断级问题`、`0 高风险安全缺陷`、`关键链路通过率 100%`。

### 1.2 推荐决策

- 推荐按照 `功能域` 而不是页面域来组织测试用例。
- 推荐把 `家长配置中心` 和 `祖辈发奖权限` 视为高风险测试对象，单独做回归。
- 推荐灰度阶段按家庭分层收集数据：`家长高参与 / 孩子高参与 / 多监护人家庭 / 祖辈参与家庭`。

### 1.3 主方案概览

| 测试层级 | 建议 | 说明 |
| --- | --- | --- |
| 单元测试 | 强制 | 覆盖规则、积分、概率、权限判定 |
| 集成测试 | 强制 | 覆盖任务到发奖全链路 |
| 安全测试 | 强制 | 覆盖篡改、防刷、重放与越权 |
| 灰度测试 | 强制 | 100 组家庭，至少 2 周观察 |

## 2. Reason：为什么这样判断

### 2.1 奖励系统最容易出现“看起来能用，但结果不可信”

如果只验证前端交互，以下风险很难发现：

- 抽奖显示正常，但后端库存重复扣减
- 积分账面正确，但流水缺失
- 祖辈按钮可点，但家庭策略并未生效
- 许愿显示通过，但奖励池未真正入库

### 2.2 灰度测试必须看亲子双侧，而不是只看家长

本产品新增模块的核心目标是：

- 孩子主动参与度提升 30%
- 家长周留存率提升 25%

因此灰度不能只看页面点击率，必须同步看：

- 孩子是否理解任务与奖励
- 家长是否觉得配置中心“可控且不麻烦”

### 4.5 灰度测试方案

#### 4.5.1 样本设计

- 总样本数：100 组家庭
- 分层建议：
  - 30 组双家长家庭
  - 20 组祖辈参与家庭
  - 30 组 6-8 岁儿童家庭
  - 20 组 9-12 岁儿童家庭

#### 4.5.2 观察周期

- 最少 2 周
- 理想为 4 周，覆盖周任务与月任务触发

#### 4.5.3 采集指标

| 指标 | 目标 |
| --- | --- |
| 孩子主动参与度 | >= 30% 提升 |
| 家长周留存率 | >= 25% 提升 |
| 乐园页进入率 | 持续观察 |
| 签到完成率 | 持续观察 |
| 任务完成率 | 持续观察 |
| 许愿提交率 | 持续观察 |
| 家长审核转化率 | 持续观察 |
| 祖辈权限开启家庭使用率 | 持续观察 |

#### 4.5.4 反馈问卷建议

- 孩子是否知道“下一步做什么”
- 孩子是否期待打开盲盒/兑换奖励
- 家长是否理解奖励边界配置
- 家长是否认为祖辈权限开关足够明确
- 家长是否信任抽奖与积分机制

### 4.6 上线门槛

- 单元测试通过率 100%
- P0 集成测试通过率 100%
- P0 安全测试通过率 100%
- 无阻断级缺陷
- 无高风险越权问题
- 核心日志链路完整
- 灰度数据达到既定目标或有明确修正方案

### 4.7 缺陷分级建议

| 等级 | 定义 |
| --- | --- |
| S1 | 奖励错误发放、越权、积分错账、库存错扣、日志缺失 |
| S2 | 审核状态错误、配置不生效、关键页面不可用 |
| S3 | 体验问题、文案歧义、动画反馈不一致 |
